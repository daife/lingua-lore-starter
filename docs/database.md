# Database Layout

[**English**](database.md) | [**中文**](database.zh.md)

---

Lingua Lore uses SQLite at two levels: one app database and one database per world.

## Storage Roots

`AppState::initialize` uses Tauri's application data directory and creates:

```text
<app-data>/
  app.db
  worlds/
    <world_id>/
      world.db
  logs/
```

The exact base path depends on the operating system and Tauri's app data directory resolution.

## App Database

Migration: `apps/desktop/src-tauri/migrations/app/001_init.sql`

| Table | Purpose |
|---|---|
| `worlds` | World library records, including title, description, storage path, target language, level, and timestamps |
| `api_profiles` | Saved model API profile with encrypted API key |
| `global_settings` | Key-value settings |

## World Database

Migration: `apps/desktop/src-tauri/migrations/world/001_init.sql`

| Table | Purpose |
|---|---|
| `world_profile` | One row describing the world premise, genre, target language, level, and narrative style |
| `characters` | Player and NPC profiles; the player character is stored as `char_player` |
| `scenes` | Scene title, location, mood, summary, current objective, and status |
| `messages` | User and assistant messages associated with turns |
| `turns` | Story turn metadata, summary, raw model JSON, and message links |
| `branch_choices` | The next choices generated by a turn; selected choices can be marked |
| `story_state` | Durable key-value state such as `scene.current`, flags, inventory, quest state, and scene fields |
| `state_update_logs` | Applied state updates with old value, new value, reason, and turn id |
| `memories` | Promoted durable memories available to read-only tools |
| `memory_candidates` | All model-proposed memory candidates, including rejected candidates |
| `relationship_state` | Current relationship values by character and dimension |
| `relationship_update_logs` | Applied relationship deltas and reasons |

The migration also drops legacy `chapters` and `world_lore` tables if they exist. The current code does not use a lore table, vocabulary table, vector database, or embedding index.

## Initialization

`create_world` inserts a row into `app.db.worlds`, creates the world directory, opens `world.db`, applies the migration, then seeds:

- `world_profile` from the create-world request.
- One `scenes` row called `Opening Scene`.
- `story_state.scene.current`, pointing at the opening scene.
- Exactly one player character, stored as `char_player`.
- Optional non-player seed characters with `trust = 0`.

## Import And Export

World export creates a ZIP containing:

```text
manifest.json
world.db
```

Import reads the manifest and `world.db`, assigns a fresh world id, updates `world_profile.id`, and inserts a corresponding `app.db.worlds` record.
